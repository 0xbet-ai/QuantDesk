# Agent Turn

How a single agent turn is executed: from a new comment, through the CLI subprocess, to the persisted session id. There is no heartbeat or scheduler — every turn is request-response triggered by a comment or a retrigger. For the MCP tool contract the agent calls during the turn, see `./MCP.md`.

## Flow

1. A new comment is posted on an experiment — by the user, by the system on desk creation, or by the server itself as a follow-up to a previous turn (e.g. the retrigger after an MCP tool finishes).
2. `triggerAgent(experimentId, role?)` loads the desk's `agentSessions` row for that role and resolves the adapter from `adapter_type`. Two adapters are registered: `claude` and `codex` (`packages/adapters/src/registry.ts`).
3. The adapter builds the CLI spawn args:
   ```bash
   # claude
   claude -p - --output-format stream-json --verbose \
     --dangerously-skip-permissions [--resume <sessionId>] \
     --mcp-config <per-turn config path>

   # codex (legacy)
   codex exec --json [resume <sessionId>] -
   ```
4. The full prompt is built by `prompt-builder.ts` and piped via stdin. Blocks are composed in this order: `analyst-system` → `tools-glossary` → `mode-{classic|realtime}` (+ `mode-generic` addendum when `desk.engine === "generic"`) → `lifecycle-rules` → `failure-escalation` (conditional) → desk / experiment / run history / memory / "new since your last turn".
5. Per-turn MCP config is generated by `buildMcpConfigForTurn` — a temp JSON file pointing at the parent server's in-process `POST /mcp` endpoint, with `X-QuantDesk-Experiment` / `X-QuantDesk-Desk` headers so tool handlers scope their side-effects to the right desk. See `./MCP.md` for the tool catalog.
6. The CLI subprocess streams JSONL events on stdout. The adapter parses them into `StreamChunk`s; the server forwards each chunk to the UI in real time and persists them in `~/.quantdesk/logs/<experimentId>.jsonl`.
7. During the turn the agent calls MCP tools via the HTTP bridge. Each tool handler runs in-process with full access to the DB, event emitter, engine adapters, and `triggerAgent`, and returns its result on the same JSON-RPC connection. The agent reads the return value or error inside the same turn and reacts without a server round-trip.
8. After the subprocess exits the server strips any stray bracketed tokens from `result.resultText` (defensive only — the legacy marker protocol is gone) and persists it as a comment authored by the current role.
9. The session id is saved to `agent_sessions.session_id` so the next turn can resume the same conversation.

## Session Management

- Sessions are scoped to **desk + role**, not experiment. The analyst retains context across experiments within the same desk; the risk manager has its own independent session.
- The prompt includes `## Currently working on Experiment #N` so the analyst stays focused on the active experiment.
- Unknown / expired session ids → the adapter retries once with a fresh session and the new id is persisted.

## Conversational approval (CLAUDE.md rule #13)

User consent for lifecycle actions lives in the same chat stream as everything else — there are no structured proposals, no approve / reject buttons, and the server renders no approval UI. The pattern is always two turns:

1. **Ask turn.** The agent describes what it would like to do in plain text, ends with a concrete question, and makes **no tool call**. The turn ends and the agent waits for the user to reply.
2. **Execution turn.** Once the user has agreed (or agreed with modifications), the agent calls the corresponding MCP tool with the final parameters in its next turn. Parameters match what was just discussed, so the tool call can never execute something the user has not seen.

```
turn N   agent: "I'd like to download 180d of BTC/USDT 1h from binance.
                 Rationale: six-month baseline for the ADX+FastD
                 momentum thesis. OK to proceed, or adjust?"
                 (no tool call — turn ends, agent waits)

user: "yes, but use ETH/USDT instead."

turn N+1 agent: calls mcp__quantdesk__data_fetch({
                    exchange: "binance",
                    pairs: ["ETH/USDT"],
                    timeframe: "1h",
                    days: 180, ...
                  })
                 tool returns { datasetId, ... }
                 agent continues the same turn with the result in context
```

Tools that need conversational approval before they fire: `data_fetch`, `request_validation`, `new_experiment`, `complete_experiment`.

Tools that fire directly without asking: `register_dataset`, `run_backtest`, `set_experiment_title`, `submit_rm_verdict`.

The server has no state machine around pending approvals. If the user never replies, the lifecycle simply pauses on the ask turn — which is fine because the ask itself is a concrete next action the user can act on (rule #12).

## No user dead-ends (CLAUDE.md rule #12)

Every lifecycle branch — including failures, rejections, and terminal turns — must surface a clear next action. When a turn ends without an automatic retrigger, the state must satisfy at least one of:

1. **The agent's last message ends with a concrete question.** The user replies in plain text and the next turn fires.
2. **A system comment names the next move.** The server posts the comment via the `systemComment(...)` helper, which forces the caller to declare `nextAction: "action" | "retrigger" | "progress"`. For `"action"`, the comment's literal content must match one of `ACTION_PHRASE_PATTERNS` so the user has something concrete to reply to.
3. **The agent is in the retrigger queue.** Another `triggerAgent` call is already scheduled (or running) for the same experiment and will produce the next turn automatically.

Silent pauses — a terminated turn with no concrete next action and no retrigger — are spec violations. Two mechanisms enforce the rule:

- **Static lint** (`server/src/__tests__/no-dead-end-lint.test.ts`) rejects any direct `createComment({ author: "system" })` outside the wrapper, and any `nextAction: "action"` call whose literal content does not contain a phrase from `ACTION_PHRASE_PATTERNS`. Runs in `pnpm test`.
- **Runtime invariant** (`server/src/services/has-next-action.ts`) is a pure function that takes a `DeskInvariantSnapshot` (pending proposal count, latest system-comment content, retrigger queue state) and returns `{ ok, reason }`. Integration tests can use the DB-touching `assertNoDeadEnd(deskId)` afterEach helper to fail a test whenever an operation leaves the desk in a silent-pause state.

The **dead-end guard** in `agent-trigger.ts` watches for bare-acknowledgment turns ("ok", "sorry", "I'll wait for guidance") that emit no MCP tool call. When the guard detects one it posts a forcing system comment and retriggers the agent once. The guard is capped per-experiment so a permanently broken agent cannot loop forever; on cap hit the user is given a multiple-choice escalation prompt.
